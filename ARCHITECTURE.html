Aero Browser How the browser opens & renders a page — file-by-file

1. View render chain

SwiftUI app. Top to bottom = outer container to actual web pixels.

AeroApp
Aero/App/AeroApp.swift
@main — app entry point. Creates the single BrowserViewModel and shows the first view in a WindowGroup.
↓
BrowserView
Aero/Features/Browser/Views/BrowserView.swift
Thin root view (~30 lines). A ZStack: background + surface + optional tab grid overlay. Attaches sheets & observers. This is the "main browser file."
↓
BrowserSurfaceView
Aero/Features/Browser/Views/BrowserSurfaceView.swift
Real layout (~220 lines). Places the web area + toolbars + address bar. Holds gestures (edge-swipe back/forward). Edit here for browser UI.
↓
WebViewRepresentable
Aero/WebKit/WebViewRepresentable.swift
Bridge SwiftUI → UIKit. Wraps Apple's WKWebView. makeUIView builds the web view, load(URLRequest) opens the page. Actual page pixels render here.
↓
WebViewCoordinator
Aero/WebKit/WebViewCoordinator.swift
Delegate. Listens to WKWebView events (load start/finish, progress, title, URL change, scroll) and pushes them back up as NavigationEvent.

2. Where a page actually opens

The line that loads a URL into the web engine:

webView.load(URLRequest(url: url))  — in WebViewRepresentable.swift, inside makeUIView (first load) and updateUIView (later navigations).

3. Flow: user types URL → page shows

  1. 1. User types in address bar → submitAddressBar() in BrowserViewModel.swift:60.
  2. 2. Text parsed (URL vs search) → resolved url.
  3. 3. tabManager.loadInActiveTab(url:) sets the active Tab's URL.
  4. 4. SwiftUI sees the Tab change → re-runs updateUIView.
  5. 5. WebViewRepresentable calls webView.load(...) → page renders.
  6. 6. WebViewCoordinator reports progress/title/URL back to the view model → UI updates.
